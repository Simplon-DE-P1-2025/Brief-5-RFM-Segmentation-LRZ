Cleaning : cascade exclusive en 4 buckets

La cascade est exclusive entre buckets, mais elle n'est pas exhaustive : le RFM v1 écarte encore les transactions anonymes et certaines anomalies. 

  raw.online_retail (1 067 371)
        │
        ├─▶ stock_movements    : description NULL ∧ customer_id NULL ∧ price = 0
        │                        ≈   4 382 lignes
        │
        ├─▶ cancellations      : invoice LIKE 'C%' (sauf stock_movements)
        │                        ≈  19 494 lignes
        │

        │                        ≈   4 805 lignes
        │
        └─▶ clean.sales        : ventes exploitables pour le RFM
                                 ≈ 802 637 lignes

Hors de ces 4 buckets, ~236 k lignes restent exclues du RFM v1, principalement des transactions anonymes (customer_id nul).

RFM : NTILE(5) + 11 segments

  -- Pour chaque client, R/F/M scores 1..5 via quintiles
  WITH ref AS (
    SELECT MAX(invoice_date)::date + 1 AS ref_date FROM clean.sales
  ),
  rfm_raw AS (
    SELECT customer_id,
           (r.ref_date - MAX(invoice_date)::date) AS recency,
           COUNT(DISTINCT invoice_id)              AS frequency,
           SUM(line_amount)                        AS monetary
    FROM clean.sales s, ref r
    GROUP BY customer_id, r.ref_date
  )
  INSERT INTO analytics.customer_rfm
  SELECT customer_id, recency, frequency, monetary,
         6 - NTILE(5) OVER (ORDER BY recency, customer_id) AS r_score,
         NTILE(5) OVER (ORDER BY frequency, customer_id)                                 AS f_score,
         NTILE(5) OVER (ORDER BY monetary, customer_id)                                  AS m_score
  FROM rfm_raw;

Le R est inversé (6 - NTILE) pour que 5 = très récent.
customer_id sert aussi de tie-breaker stable pour éviter les changements de score sur les ex aequo d'un rerun à l'autre. Les 11 segments business sont assignés ensuite via un UPDATE CASE ordonné spécifique → général.

Choix techniques justifiés

  • Postgres-first : tout le typage est défini dans le DDL (customer_id INTEGER NULL) → pandas ne casse pas
  • NTILE(5) en SQL plutôt qu'en Python : plus rapide, plus lisible, et profite directement des index
  • Cascade exclusive au lieu d'une seule WHERE : chaque ligne brute est traçable dans son bucket
  • Airflow PythonOperator + .sql files : les SQL sont versionnés indépendamment, le DAG les enchaîne via un wrapper psycopg2 minimal
  • Flask + reveal.js : Streamlit ne sait pas faire de présentation slide-by-slide avec charts interactifs. reveal.js + Plotly oui. C'est ce qui permet cette démo.
  • uv : 10× plus rapide que pip pour les installs

Apprentissages

  • Le piège du typage pandas qui force customer_id en float64 et coupe les jointures — résolu en castant côté Postgres
  • L'ordre des CASE WHEN compte : du plus spécifique au plus général, sinon les sous-segments sont absorbés
  • Airflow + idempotence : chaque task TRUNCATE en début, le DAG est rejouable sans état préalable
  • Les vues SQL sont parfaites pour l'enrichissement (segment_label A→K, macro_segment) sans toucher au DDL initial

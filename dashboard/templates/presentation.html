{# ───── S7. Dockerfile décortiqué en couches ─ 1 min 30 ───── #}

Dockerfile — la pile de couches

Chaque instruction = un layer Docker cacheable. L'ordre des COPY est capital pour le cache.

FROM python:3.11-slim
Base légère (~45 MB) — pas de compilateurs, pas de man pages
COPY --from=astral-sh/uv /uv /usr/local/bin/uv
Injection du binaire uv — 10× plus rapide que pip
RUN groupadd etlgroup && useradd etluser
User non-root — sécurité : un container root compromis = hôte compromis
COPY pyproject.toml uv.lock ./
Manifest d'abord — si le code change mais pas les deps, Docker réutilise le layer d'install
RUN uv sync --frozen --no-dev
Install déterministe depuis le lockfile — --frozen = pas de résolution
COPY etl/ ./etl/
Code applicatif — APRÈS les deps, donc modifiable sans invalider le cache des layers du haut
COPY data/ ./data/
Dataset source (on l'embarque dans l'image pour une démo 100 % offline)
USER etluser
Bascule non-root à l'exécution — l'install a été faite en root, l'exécution non

{# ───── S10. NTILE(5) ─ 1 min ───── #}

RFM : NTILE(5) en SQL pur

  WITH ref AS (
    SELECT MAX(invoice_date)::date + 1 AS ref_date FROM clean.sales
  ),
  rfm_raw AS (
    SELECT customer_id,
           (r.ref_date - MAX(invoice_date)::date) AS recency,
           COUNT(DISTINCT invoice_id)              AS frequency,
           SUM(line_amount)                        AS monetary
    FROM clean.sales s, ref r
    GROUP BY customer_id, r.ref_date
  )
  INSERT INTO analytics.customer_rfm
  SELECT customer_id, recency, frequency, monetary,
         6 - NTILE(5) OVER (ORDER BY recency, customer_id) AS r_score,
         NTILE(5) OVER (ORDER BY frequency, customer_id) AS f_score,
         NTILE(5) OVER (ORDER BY monetary, customer_id)  AS m_score
  FROM rfm_raw;

Le R est inversé (6 - NTILE) pour que 5 = très récent. customer_id sert de tie-breaker stable — garantit des scores identiques d'un rerun à l'autre sur les ex aequo.

{# ───── S12a. Sous-slide : piège pandas customer_id ───── #}
Pendant que ça tourne · 1/3

Le piège pandas customer_id

  # Ce que fait pandas par défaut
  df = pd.read_excel("online_retail_II.xlsx")
  df["Customer ID"].dtype
  # → dtype('float64')   ← 12345.0 au lieu de 12345

  # Conséquence : la jointure avec n'importe quelle table
  # qui a customer_id INTEGER plante silencieusement.

Pourquoi : pandas caste les colonnes numériques contenant des NaN en float64 (les entiers n'admettent pas NaN en numpy).

Notre fix : customer_id INTEGER NULL dans le DDL Postgres → COPY FROM force le cast à l'entrée. Pandas ne voit jamais la colonne — et ne casse donc rien.

{# ───── S12b. Sous-slide : ordre des CASE WHEN ───── #}
Pendant que ça tourne · 2/3

L'ordre des CASE WHEN

  -- ❌ Ce qui absorbe les sous-segments
  CASE
    WHEN r >= 4 AND f <= 3 THEN 'Potential Loyalist'   ← trop large en premier
    WHEN r = 5 AND f = 1    THEN 'New Customers'        ← jamais atteint
    ...

  -- ✅ Spécifique → général
  CASE
    WHEN r = 5 AND f = 1       THEN 'New Customers'
    WHEN r >= 4 AND f <= 2 AND m <= 2 THEN 'Promising'
    WHEN r >= 4 AND f <= 3    THEN 'Potential Loyalist'
    ...

Sans le fix on n'avait que 9/11 segments peuplés : les 50 New Customers et les 364 Promising étaient absorbés par Potential Loyalist. Un CASE SQL court-circuite au premier match, donc l'ordre est sémantique, pas cosmétique.

{# ───── S12c. Sous-slide : idempotence Airflow ───── #}
Pendant que ça tourne · 3/3

Idempotence Airflow

  def task_transform_clean_rfm():
      _truncate_downstream()          # ← TRUNCATE en début de task
      _run_sql_file("02_clean.sql")
      _run_sql_file("03_rfm.sql")

  # Chaque task repart d'un état connu.
  # → on peut relancer le DAG autant de fois qu'on veut
  # → pas de doublons, pas d'état résiduel

Pourquoi ça compte : en production, une task Airflow échoue et retry. Sans idempotence, un retry peut doubler les insertions. Avec TRUNCATE au début, on garantit que l'état final est toujours le même, quel que soit le chemin.

La seule table qu'on ne TRUNCATE pas : raw.online_retail. ingest_xlsx fait son propre TRUNCATE via truncate=True.

Choix techniques justifiés

🐳

Docker Compose

Un seul YAML, 6 services, reproductible partout. Pas d'install système.

⚡

uv

Gestion Python 10× plus rapide que pip, lockfile déterministe.

🎯

Postgres-first

Typage dans le DDL. customer_id INTEGER NULL — pandas ne casse pas.

📊

NTILE(5) en SQL

Plus rapide que pandas, plus lisible, profite des index Postgres.

🔄

Python + .sql séparés

Les SQL sont versionnés indépendamment, le DAG les enchaîne via un wrapper psycopg2 minimal.

🎥

Flask + reveal.js

Streamlit ne fait pas de slides plein écran avec charts interactifs. reveal.js oui.

Ce qu'on retient
